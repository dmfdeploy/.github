# Support

*This is the org-wide default; a repo's own SUPPORT.md, if present, wins.*

- **Questions** ("how do I…", "is this expected…") →
  [umbrella Discussions, Q&A](https://github.com/dmfdeploy/dmfdeploy/discussions).
- **Bugs and feature requests** → the canonical backlog is
  [umbrella Issues](https://github.com/dmfdeploy/dmfdeploy/issues), even for
  work whose code lives in a component repo (that's what the `component:*`
  labels are for). Filing in a component repo is fine too — those are treated
  as drive-by intake and triaged into the umbrella backlog within about a week.
- **Architecture proposals** → start an RFC thread in
  [Discussions](https://github.com/dmfdeploy/dmfdeploy/discussions); the
  RFC-before-ADR pipeline is described in
  [CONTRIBUTING](https://github.com/dmfdeploy/dmfdeploy/blob/main/CONTRIBUTING.md).
- **Security vulnerabilities** → never a public issue; use GitHub's private
  vulnerability reporting on the affected repo (see its SECURITY.md).

The full statement of how work is tracked here:
[docs/WORKING-MODEL.md](https://github.com/dmfdeploy/dmfdeploy/blob/main/docs/WORKING-MODEL.md).
